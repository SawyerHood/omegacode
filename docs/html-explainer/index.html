01 / Mental model

Why structure work across agents?

One agent has one context window and one perspective. A workflow buys you three things a single context can't — and the file is where you encode that structure.

REACH

Be comprehensive

Decompose the task and cover it in parallel — one reader per subsystem, one finder per review dimension, one searcher per modality. Nothing waits on anything it doesn't depend on.

TRUST

Be confident

Independent perspectives and adversarial checks before committing. Spawn skeptics that try to refute each finding; only what survives a panel ships. Cross-provider disagreement is signal.

SCALE

Beyond one context

Migrations, audits, broad sweeps — work too large for any single context window. Each agent holds one slice; the workflow holds the shape and stitches results back together.

The right move is often hybrid: scout first (list the files, scope the diff) to discover the work-list, then write the workflow to pipeline over it. For larger work, chain several workflows — Understand → Design → Review — reading each result before deciding the next. You stay in the loop; each run is one well-scoped fan-out.

03 / Concurrency

pipeline() vs the barrier

The single most important authoring decision. Both runs below do identical work — five items, three stages, the same durations. pipeline() lets each item flow ahead independently; parallel() between stages is a barrier where every item waits for the slowest sibling. Watch them race.

t = 0.0s
find verify report waiting at barrier

pipeline(items, find, verify, report)

no barrier — each item proceeds the moment its previous stage ends

parallel(find) → parallel(verify) → parallel(report)

barrier after each stage — everyone waits for the slowest item

Default to pipeline(). A barrier is correct only when stage N genuinely needs cross-item context from all of stage N−1:

  • Dedup / merge across the full result set before expensive downstream work
  • Early-exit when the total is zero — “0 bugs found → skip verification entirely”
  • Stage N's prompt references “the other findings” for comparison

Not justified by “I need to flatten first” (do it inside a stage), “the stages are conceptually separate” (that's exactly what pipeline models), or “it's cleaner code” (barrier latency is real). When in doubt: pipeline.

04 / The DSL

Eight hooks. That's the whole language.

The script body is plain JavaScript with these injected. No filesystem, no network, no imports — the agents do the I/O; the file only does orchestration.

05 / Providers

Every agent() runs on a real provider

Default: set neither provider nor model. Each agent inherits whatever the run was invoked with (--provider / --model, default codex). Pin one only when the user asked for it, or a step deliberately needs cross-provider diversity. omegacode doctor shows what's installed and authed.

codex

OpenAI · gpt-5.x · the default
  • Auth: the codex CLI, ChatGPT login — ignores OPENAI_API_KEY
  • Structured output: free-form working turn, then a schema-constrained extraction turn
  • Extras: built-in hosted tools incl. image generation come from that auth

claude-code

Anthropic · via Claude Agent SDK

  • Structured output: delivered on the final result only — intermediate messages stay free-form
  • Effort: max is Opus-tier; unsupported levels silently downgrade

06 / Determinism & resume

Every run is a journal you can replay

Runs persist to ~/.omegacode/runs/<id>/ and print a runId. Re-run with --resume <runId> and the longest unchanged prefix of agent() calls returns cached results instantly — only the first edited call and everything after it runs live. Pick a scenario:

Determinism is why Date.now(), Math.random() and argless new Date() throw inside a workflow (and are rejected by a submit-time lint) — they'd make replays diverge. Use the journal-seeded now() / random() instead, or pass timestamps in via args.
07 / Quality patterns

Shapes worth stealing

Pick by task, compose freely — and invent new harnesses when the task calls for it (tournament brackets, self-repair loops, staged escalation). Scale to the ask: “find any bugs” → a few finders, single-vote verify; “thoroughly audit this” → larger pool, 3–5-vote adversarial pass, synthesis stage.

09 / CLI & limits

The command surface

100
concurrent agents per workflow (default) — override with --concurrency N; excess calls queue
1,000
lifetime agent cap per run — a runaway-loop backstop, far above any real workflow
4,096
max items per single parallel()/pipeline() call — exceeding is an explicit error, never silent truncation
read-only
the default sandbox — grant workspace-write only when an agent must write
The live viewer auto-starts on run (reused if already up) and idle-shuts-down when nothing's active. It reads ~/.omegacode/runs, streams via SSE, and never executes anything: run list, live phase/agent tree, per-agent chat-feed drilldown. The moment a run starts it prints view: http://127.0.0.1:4123/#/run/<id> — when an agent runs a workflow for you, it should surface that link before the run finishes so you can watch live. --fake smoke-tests with no real agents; --json returns {runId, status, url, result, error}.
